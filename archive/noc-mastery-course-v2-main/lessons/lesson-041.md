# Lesson 41: SIP Headers — The Essential Ones and What They Tell You
**Module 1 | Section 1.10 — SIP Protocol Deep Dive**
**⏱ ~10 min read | Prerequisites: Lesson 38**

---

Being able to read SIP headers fluently is the single most valuable skill for a telecom NOC engineer. Every SIP message is a story — the headers tell you where the message came from, where it's going, who the call is for, and what path it took. This lesson is your header-by-header field guide.

## The Via Header — Tracing the Request Path

Via is arguably the most important header for troubleshooting. Each SIP entity that forwards a request **adds a Via header at the top** of the stack. Responses follow the Via headers in reverse to get back to the sender.

```
Via: SIP/2.0/UDP proxy2.telnyx.com:5060;branch=z9hG4bK-2;received=10.10.5.20
Via: SIP/2.0/UDP proxy1.telnyx.com:5060;branch=z9hG4bK-1;received=10.10.1.10
Via: SIP/2.0/UDP 192.168.1.50:5060;branch=z9hG4bKnashds7;rport=5060;received=203.0.113.50
```

**Reading this:** The request originated from 192.168.1.50 (a private IP), was received by proxy1 from public IP 203.0.113.50 (NAT detected — `received` parameter shows the actual source IP), then forwarded to proxy2.

**Key parameters:**
- **branch:** Transaction identifier. Must start with `z9hG4bK` (magic cookie indicating RFC 3261 compliance). Each transaction gets a unique branch.
- **received:** Added by the receiving proxy when the source IP differs from the address in the Via. Critical for NAT detection.
- **rport:** Requested by the client (RFC 3581) — tells the proxy to record the actual source port. Essential for NAT traversal.
- **transport:** UDP, TCP, TLS, or WSS (WebSocket Secure).

🔧 **NOC Tip:** When debugging connectivity issues, the Via `received` parameter tells you the actual public IP of the sender. If this differs from the Contact or From addresses, you're dealing with NAT. Compare `received` values across hops to trace the real network path.

## From and To — Dialog Identification (Not Routing!)

```
From: "Alice" <sip:+15559876543@customer.com>;tag=1928301774
To: <sip:+15551234567@sip.telnyx.com>;tag=a6c85cf
```

**Common misconception:** From and To are NOT used for routing. They identify the dialog participants.

- **From:** Identifies the initiator. The `tag` parameter is generated by the UAC and uniquely identifies this side of the dialog.
- **To:** Identifies the target. The `tag` parameter is added by the UAS in the response. Together, Call-ID + From-tag + To-tag uniquely identify a dialog.

The From URI can be different from the actual caller identity. For caller ID purposes, P-Asserted-Identity is more authoritative.

## Contact — Direct Reachability

```
Contact: <sip:+15559876543@203.0.113.50:5060;transport=udp>
```

Contact provides a **direct address** for reaching this UA. Unlike From/To (which are logical identities), Contact is a physical address — "send subsequent requests here."

**Why it matters:** Mid-dialog requests (re-INVITE, BYE) are sent to the Contact address, not through proxies (unless Record-Route is used). If the Contact address is wrong (NAT-mangled private IP), BYE never reaches the endpoint → ghost call.

**In a B2BUA context:** Telnyx rewrites the Contact header on both legs. The customer sees Telnyx's address; the carrier sees Telnyx's address. Neither sees the other directly.

## Call-ID — Unique Call Identifier

```
Call-ID: f81d4fae-7dec-11d0-a765-00a0c91e6bf6@customer.com
```

Globally unique identifier for this call (dialog). Generated by the UAC. Every SIP message in the same call carries the same Call-ID.

🔧 **NOC Tip:** Call-ID is your primary search key in Graylog and SIP trace tools. When investigating a specific call, get the Call-ID first, then query all messages with that Call-ID. In a B2BUA, the customer-side Call-ID and carrier-side Call-ID will be **different** — you need to find the correlation in Telnyx's systems.

## CSeq — Command Sequence

```
CSeq: 314159 INVITE
```

Contains a sequence number and the method name. The sequence number increments with each new request within a dialog (INVITE = 1, re-INVITE = 2, BYE = 3). Retransmissions keep the same CSeq.

**Why it matters:** CSeq + method identifies a specific transaction within a dialog. The method in CSeq is essential because ACK uses the same CSeq as the INVITE it acknowledges.

## Record-Route and Route — Keeping Proxies in the Path

```
Record-Route: <sip:proxy.telnyx.com;lr>
```

When a proxy adds Record-Route, it's saying "I want to stay in the signaling path for the entire dialog." Without Record-Route, subsequent requests (BYE, re-INVITE) go directly between endpoints. With Record-Route, they pass through the proxy.

**Route header:** The inverse of Record-Route. Built from the Record-Route set, Route tells a request which proxies to traverse. It's the "pre-computed path."

**`lr` parameter:** "loose routing" — RFC 3261 compliant. Without `lr`, the proxy uses strict routing (deprecated, but some legacy systems still use it, causing routing headaches).

🔧 **NOC Tip:** If BYE messages aren't reaching one side of a call, check the Route set. If Record-Route was used during setup but the subsequent BYE doesn't include the correct Route headers, the BYE goes directly to the Contact address, potentially bypassing firewalls or NAT that only allow traffic through the proxy.

## P-Asserted-Identity — Trusted Caller ID

```
P-Asserted-Identity: "Alice Smith" <sip:+15559876543@telnyx.com>
```

Carries the authenticated identity of the caller within a trusted network. Unlike From (which can be set to anything), P-Asserted-Identity is only added by entities that have verified the caller's identity.

**Privacy interaction:** If the caller requests privacy (`Privacy: id`), the P-Asserted-Identity is removed before the message leaves the trusted network.

In Telnyx's context: customers provide caller ID in their INVITE. Telnyx validates it against their account's allowed numbers and sets P-Asserted-Identity on the carrier leg. This prevents caller ID spoofing.

## Max-Forwards — Loop Prevention

```
Max-Forwards: 70
```

Decremented by each proxy. If it reaches 0, the proxy responds 483 Too Many Hops. Default value is 70. This prevents infinite routing loops.

## Content-Type and Content-Length — Body Description

```
Content-Type: application/sdp
Content-Length: 312
```

Most INVITE requests and responses carry an SDP body. But SIP can carry other content types: `multipart/mixed` (SDP + other data), `application/dtmf-relay` (SIP INFO DTMF), `message/sipfrag` (in NOTIFY for REFER progress).

## Custom and Extension Headers

### X-Headers (Vendor-Specific)
```
X-Telnyx-CallId: v3:abc123def456
```
Vendor-specific headers for passing proprietary information. Telnyx uses custom headers for internal call correlation, feature flags, and debugging information.

### Diversion (RFC 5806)
```
Diversion: <sip:+15551112222@telnyx.com>;reason=unconditional;counter=1
```
Indicates call forwarding history — the number that diverted the call and why. Used for call forwarding tracing.

### History-Info (RFC 7044)
```
History-Info: <sip:+15551112222@telnyx.com>;index=1
History-Info: <sip:+15559998888@telnyx.com>;index=1.1
```
Alternative to Diversion for recording call routing history. More structured but less widely implemented.

### Supported / Require / Proxy-Require

**Supported:** Lists optional SIP extensions the UA supports.
**Require:** Lists extensions the UA **requires** — if the other side doesn't support them, it must reject the request with 420 Bad Extension.
**Proxy-Require:** Same as Require, but for proxies.

```
Supported: timer, replaces, 100rel
Require: 100rel
```

## Reading a Complete SIP Message

Let's put it all together with a real INVITE:

```
INVITE sip:+15551234567@sip.telnyx.com SIP/2.0
Via: SIP/2.0/UDP 192.168.1.50:5060;branch=z9hG4bK776;rport
Max-Forwards: 70
From: "Sales Dept" <sip:+15559876543@customer.com>;tag=abc123
To: <sip:+15551234567@sip.telnyx.com>
Contact: <sip:+15559876543@192.168.1.50:5060>
Call-ID: 98765@192.168.1.50
CSeq: 1 INVITE
Allow: INVITE, ACK, BYE, CANCEL, OPTIONS, REFER, NOTIFY
Supported: replaces, timer
Content-Type: application/sdp
Content-Length: 246
```

**What this tells us:**
- Originated from a device at 192.168.1.50 (private IP — behind NAT)
- Calling +15551234567 via Telnyx
- Caller ID shows "Sales Dept" at +15559876543
- Contact is the private IP (NAT problem — BYE responses will try to reach 192.168.1.50)
- Supports REFER (can handle transfers) and timer (session timer support)
- rport requested — wants the proxy to tell it its public port

## Real Troubleshooting Scenario: Caller ID Showing Wrong Number

**Symptom:** Customer's outbound calls show a different caller ID than expected.

**Investigation:**
1. Pull SIP trace for the A-leg (customer → Telnyx)
2. From header shows: `From: <sip:+15559876543@customer.com>`
3. P-Asserted-Identity not present in customer's INVITE
4. Telnyx's outbound voice profile is configured with a default caller ID of +15551111111
5. B-leg INVITE shows: `From: <sip:+15551111111@telnyx.com>`

**Root cause:** The customer's PBX is sending the caller ID in the From header, but Telnyx's outbound profile overrides it with the configured default because P-Asserted-Identity wasn't present.

**Resolution:** Configure the customer's PBX to send caller ID in P-Asserted-Identity, or adjust the Telnyx outbound voice profile to use the From header as caller ID source.

---

**Key Takeaways:**
1. Via headers trace the request path — `received` and `rport` parameters reveal NAT and the actual source IP
2. From/To identify dialog participants (not routing!) — the tag parameters + Call-ID uniquely identify a dialog
3. Contact provides the direct address for mid-dialog requests — NAT-mangled Contact causes ghost calls and BYE failures
4. Call-ID is your primary search key for debugging — remember that B2BUAs use different Call-IDs on each leg
5. Record-Route keeps proxies in the signaling path for the entire dialog — without it, BYE goes directly to Contact
6. P-Asserted-Identity carries trusted caller ID — distinct from From, which can be set arbitrarily
