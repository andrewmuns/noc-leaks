## Why State Management Matters

SIP is fundamentally a *stateful* protocol, but its state is distributed across multiple entities. Unlike the PSTN where a central switch owns the call state, SIP distributes state between endpoints and intermediaries. When this state gets out of sync — when one side thinks a call exists but the other doesn't — you get ghost calls, hung sessions, and billing errors. Understanding transactions and dialogs is the key to preventing and diagnosing these problems.

---

## Transactions: The Atomic Unit

A **transaction** is a single request-response exchange. It's the smallest unit of SIP interaction. Every SIP request creates a transaction that lives until a final response (2xx-6xx) is received.

### Transaction Identification

A transaction is uniquely identified by:
- The **Via branch parameter** (a unique token the client generates)
- The **CSeq method** (because a CANCEL and its INVITE share a Via branch but are separate transactions)

When a response arrives, the transaction layer matches it to the original request using these identifiers. If no match is found, the response is dropped — this is a common source of "response not processed" bugs when SIP ALGs or intermediaries rewrite Via headers incorrectly.

### Client vs. Server Transactions

The entity that sends the request runs a **client transaction**. The entity that receives and responds runs a **server transaction**. A proxy runs *both* — a server transaction for the incoming request and a client transaction for the forwarded request.

### INVITE vs. Non-INVITE Transactions

SIP has two types of transactions with fundamentally different state machines:

**Non-INVITE transactions** (REGISTER, BYE, OPTIONS, etc.) are simple:
- Request → Final Response. Done.
- Timer F = 32 seconds. If no response, transaction fails with 408.

**INVITE transactions** are complex because calls take time to set up:
- INVITE → Provisional (1xx) → Final Response → ACK
- Timer B = 32 seconds for the initial response
- The ACK is part of the INVITE transaction for *error* responses (4xx-6xx) but forms a *separate* transaction for 2xx (success)

This asymmetry is one of the most confusing aspects of SIP. Why? Because a 2xx to an INVITE can be retransmitted by the UAS (it's critical — it means "answer the call"), and the ACK must be generated by the UAC, not the transaction layer. For error responses, the transaction layer handles ACK automatically.

🔧 **NOC Tip:** If you see repeated 200 OK responses in a trace without a corresponding ACK, the ACK isn't reaching the UAS. This causes the UAS to retransmit the 200 OK until it gives up and tears down the call. Check NAT/firewall rules and Record-Route paths.

## Dialogs: The Long-Lived Relationship

A **dialog** is a peer-to-peer SIP relationship that persists for the duration of a call (or subscription). While transactions last seconds, dialogs last minutes or hours.

### Dialog Identification

A dialog is uniquely identified by three values:
- **Call-ID**: Globally unique identifier generated by the caller
- **From-tag**: Tag parameter in the From header (set by the caller)
- **To-tag**: Tag parameter in the To header (set by the callee in the response)

The dialog doesn't fully exist until the To-tag is assigned — which happens in the first response that carries one (typically 180 Ringing or 200 OK). Before the To-tag, it's a "half-dialog" or **early dialog**.

### Dialog State Machine

```
    INVITE sent
         │
         ▼
    ┌─────────┐     1xx with To-tag
    │  Trying  │ ──────────────────► Early Dialog
    └─────────┘                         │
                                        │ 2xx
                                        ▼
                                  ┌───────────┐
                                  │ Confirmed  │ ◄── Call is active
                                  └───────────┘
                                        │
                                        │ BYE
                                        ▼
                                  ┌───────────┐
                                  │ Terminated │
                                  └───────────┘
```

### What a Dialog Stores

Once established, a dialog maintains:
- **Route set**: The path mid-dialog requests must follow (from Record-Route headers)
- **Remote target**: The Contact URI of the other party
- **Local/remote sequence numbers**: CSeq tracking for each direction
- **Call-ID, tags**: The identifying triple

This stored state is why mid-dialog requests (re-INVITE, BYE) can reach the right destination — they use the route set and remote target, not DNS.

## The Relationship Between Transactions and Dialogs

A single dialog contains multiple transactions over its lifetime:

1. **INVITE transaction** — creates the dialog
2. **Re-INVITE transactions** — modify the session (hold, codec change)
3. **BYE transaction** — terminates the dialog
4. **Other in-dialog transactions** — INFO, REFER, UPDATE

Each transaction is independent but operates *within* the dialog context. The dialog provides the routing information; the transaction provides the request-response exchange.

## Forking: One INVITE, Multiple Dialogs

When a proxy receives an INVITE for a user registered on multiple devices, it can **fork** — sending the INVITE to all registered contacts simultaneously (parallel forking) or sequentially (serial forking).

Each device that responds with a To-tag creates a separate **early dialog**. When one device answers (200 OK), that dialog becomes confirmed, and the others receive CANCEL.

Forking creates a tricky situation: the caller has one INVITE transaction but potentially multiple early dialogs. The caller must be prepared to receive multiple provisional responses with different To-tags and handle the ACK/BYE correctly for each.

🔧 **NOC Tip:** If you see a call that rings multiple devices but creates duplicate CDRs or billing anomalies, check for forking issues — specifically, whether CANCELs are properly sent to the non-answered devices after one device picks up.

## Ghost Calls and Hung Sessions

When dialog state gets out of sync, bad things happen:

### Ghost Calls
A "ghost call" exists in one system's records but not in reality. Common causes:
- **BYE lost in transit**: One side sends BYE, it doesn't arrive, the other side thinks the call is still active. The session timer (Lesson 47) is the safety net for this.
- **ACK lost for 200 OK**: The callee retransmits 200 OK, gives up, tears down locally. The caller thinks the call was established.
- **B2BUA leg mismatch**: In Telnyx's B2BUA architecture, if one leg tears down but the internal correlation fails, the other leg may remain active.

### Hung Sessions
A session that both sides have "forgotten" but still consumes resources (media ports, memory) on intermediary systems. Common in B2BUA deployments when the cleanup routine fails.

### Prevention and Detection
- **Session timers** (RFC 4028): Periodic re-INVITE/UPDATE to confirm both sides are still alive
- **OPTIONS ping**: Sending OPTIONS to verify the remote endpoint is still there
- **CDR reconciliation**: Comparing call duration between legs to detect mismatches
- **Grafana alerts**: Monitor for sessions exceeding maximum expected duration

## Real-World Troubleshooting Scenario

**Problem:** Customers report being billed for calls that lasted seconds, but their CDRs show multi-hour durations.

**Investigation:**
1. Pull the SIP trace for the call. You find: INVITE → 200 OK → ACK → (media flows for 30 seconds) → then nothing. No BYE from either side.
2. The customer's PBX crashed mid-call and never sent BYE.
3. Telnyx's B2BUA kept the session alive because it had no indication the call ended.
4. Eventually, a session timer fired (after the configured interval — say 30 minutes) and tore down the call.

**Fix:** Ensure session timers are configured with reasonable intervals (e.g., 90 seconds). Monitor for sessions that exceed typical maximum call duration. Consider enabling RTP timeout — if no media flows for N seconds, tear down the session.

## Stateful vs. Stateless Proxies

A **stateful proxy** maintains transaction state — it matches responses to requests and can perform forking, retransmission, and CANCEL processing. Most SIP proxies in telecom environments are stateful.

A **stateless proxy** forwards requests and responses without maintaining state. It's faster and scales better but can't fork, can't match responses, and can't generate CANCEL. Used for simple load balancing or message routing.

A **B2BUA** (like Telnyx) maintains *full dialog state* for both legs. It's the most resource-intensive but provides the most control — it can modify headers, SDP, manage media, and bill accurately.

---

**Key Takeaways:**
1. A **transaction** is a single request-response exchange; a **dialog** is a long-lived relationship spanning multiple transactions
2. Dialogs are identified by the triple: Call-ID + From-tag + To-tag
3. INVITE transactions have a special three-way handshake (INVITE → 200 → ACK) where ACK for 2xx is *not* part of the transaction
4. Forking creates multiple early dialogs from a single INVITE — proper CANCEL handling is essential
5. Ghost calls occur when dialog state gets out of sync — session timers and RTP timeouts are critical safety nets
6. B2BUAs maintain full dialog state for both call legs, enabling features but requiring robust state management
7. Missing ACKs cause 200 OK retransmission storms — always check ACK delivery when calls drop shortly after answer
