# Lesson 127: Reading SIP Traces — A Systematic Approach

**Module 4 | Section 4.2 — SIP Debugging**
**⏱ ~9 min read | Prerequisites: Lessons 37-47**

---

SIP traces are the single most powerful tool for debugging call failures. A complete SIP trace reveals the entire call flow — what was requested, what was negotiated, where it failed, and why. But SIP traces can be thousands of lines long and cryptic at first glance. This lesson provides a systematic approach to read them efficiently.

## Finding SIP Traces

At Telnyx, SIP traces are logged to Graylog and indexed by:

- **Call-ID**: The unique identifier for a SIP dialog (Lesson 41)
- **Source/Destination number**: The phone numbers involved
- **Timestamp**: When the call occurred
- **Connection ID**: The Telnyx connection handling the call
- **Customer credential**: If using credential authentication

**Graylog queries:**
```
# By Call-ID
type:SIP AND Call-ID:abc123-def456

# By phone number
type:SIP AND (From:sip:+15551234567@ OR To:sip:+15559876543@)

# By time range and connection
type:SIP AND connection_id:conn-abc123 AND timestamp:["2026-02-22 14:00" TO "2026-02-22 14:05"]

# Failed calls only
type:SIP AND ("SIP/2.0 4" OR "SIP/2.0 5" OR "SIP/2.0 6")
```

🔧 **NOC Tip:** Customer reports with a Call-ID are gold. The Call-ID is generated by the caller's device and persists across the entire call. When customers report issues, always ask: "Do you have the Call-ID or a packet capture?" This 32-character string lets you jump directly to the relevant logs instead of searching by phone number and time.

## The Call Flow Reconstruction

SIP traces are interleaved messages from multiple legs in the B2BUA architecture (Lesson 37). Telnyx's sip-proxy:

```
Customer → SIP Proxy (Customer Leg) → SIP Proxy (Carrier Leg) → PSTN Carrier
              ↑                          ↑
          Logs tagged                 Logs tagged
          "inbound"                  "outbound"
```

The trace shows messages on both legs, often interleaved. You must mentally separate them:

```
# Customer Leg (inbound)
Inbound INVITE from customer
Inbound ACK from customer
Inbound BYE from customer

# Carrier Leg (outbound)
Outbound INVITE to carrier
Outbound 200 OK from carrier
Outbound BYE to carrier
```

## Systematic Trace Reading: The Checklist

When examining a failed call, follow this order:

### Step 1: Identify the Failure Point

Scan for error response codes:
```
SIP/2.0 408 Request Timeout
SIP/2.0 480 Temporarily Unavailable
SIP/2.0 503 Service Unavailable
SIP/2.0 486 Busy Here
```

Find the **first** error in the flow. Often later errors are cascades from an earlier failure.

### Step 2: Which Leg Failed?

Look at the message context around the error:
- If the error is on the **customer leg**, Telnyx is rejecting the call or the customer is failing to respond
- If the error is on the **carrier leg**, an upstream carrier is rejecting the call

```
# This means the INVITE was forwarded to a carrier, but they responded 503
Outbound INVITE → carrier
Inbound 503 ← carrier  ← Problem is carrier-side
```

### Step 3: Read the Request That Triggered the Error

Look backward to what caused the rejection:

```
INVITE sip:+15559876543@telnyx.com SIP/2.0
Via: SIP/2.0/UDP 10.0.1.5:5060
From: "Caller" <sip:+15551234567@customer.com>
To: <sip:+15559876543@telnyx.com>
Call-ID: abc123-def456@customer.com
CSeq: 1 INVITE
Contact: <sip:customer-ua@10.0.1.5:5060>
Content-Type: application/sdp
Content-Length: 234

v=0
o=- 12345 12345 IN IP4 10.0.1.5
s=-
c=IN IP4 10.0.1.5  ← Private IP!
t=0 0
m=audio 10000 RTP/AVP 0 8
...
```

🔧 **NOC Tip:** The Contact header and SDP connection address (c=) reveal private IPs — the NAT problem (Lessons 26-28). If these contain 10.x.x.x, 192.168.x.x, or 172.16-31.x.x and the customer is outside the network, NAT is unhandled. Look at the responses — did Telnyx try to send RTP to that private IP (fail) or did SDP manipulation happen (possibly successful)?

### Step 4: Check Authentication and Authorization

For 401 or 403 responses:

```
# 401 Unauthorized
SIP/2.0 401 Unauthorized
WWW-Authenticate: Digest realm="telnyx.com", nonce="abc123"

# Customer should resend with Authorization header
Authorization: Digest username="customer123", realm="telnyx.com", 
  nonce="abc123", response="hash-of-password"
```

If you see:
- 401, then no subsequent request with Authorization → Customer not sending credentials
- 401, then Authorization header, then another 401 → Wrong password
- 403 Forbidden → Credentials valid, but not authorized for this action

### Step 5: Examine SDP Negotiation

For 488 Not Acceptable Here or no audio after 200 OK:

```
# Customer's offer
m=audio 10000 RTP/AVP 0 18 8  
# Codecs: G.711 ulaw (0), G.729 (18), G.711 alaw (8)

# Telnyx's answer
m=audio 20000 RTP/AVP 0 101
# Accepting: G.711 ulaw (0), telephone-event (101)
# Notice: G.729 (18) was offered but not accepted
```

Codec negotiation failures:
- If no codecs intersect → 488 response
- If customer can only do G.729 but Telnyx configured for G.711 only → 488

🔧 **NOC Tip:** For one-way audio issues, compare the SDP in both directions. Customer sends RTP to the IP:port from the 200 OK's SDP (the answer). If they're sending to the wrong IP or port, check the Contact and SDP manipulation. The send address (what customer sends TO) and recv address (what customer listens ON) are different. Customer RECVs on their advertised port, SENDS to our advertised port.

## Common Trace Patterns

### Pattern 1: Immediate 408 Timeout

```
Customer INVITE → Telnyx
60 seconds pass...
Telnyx 408 Request Timeout → Customer
```

Likely causes:
- Telnyx couldn't reach carrier (carrier down, routing issue)
- Carrier never responded to INVITE
- Carrier responded but response never reached Telnyx (network issues)

### Pattern 2: 486 Busy After 180 Ringing

```
Customer INVITE → Telnyx → Carrier
180 Ringing ← Carrier
180 Ringing ← Telnyx ← Customer
480 Temporarily Unavailable ← Carrier (after 10 seconds)
480 ← Telnyx ← Customer
```

The destination is busy or unavailable. This is typically normal user behavior, but if happening for all calls, the destination might be having issues.

### Pattern 3: No ACK After 200 OK

```
Customer INVITE → Telnyx → Carrier
180 Ringing ←
200 OK from Carrier → Telnyx
200 OK → Customer
(ACK never arrives from Customer)
Telnyx BYE (timeout) → Carrier  (sends BYE since customer never ACKed)
```

The customer didn't send ACK. This is often a NAT/firewall issue — the 200 OK reached the customer, but their ACK can't traverse back to Telnyx.

### Pattern 4: Re-INVITE During Call (On Hold)

```
# Mid-call
Customer re-INVITE (SDP with a=sendonly) → Telnyx  ← Putting on hold
200 OK ← Telnyx → Customer
ACK → Telnyx
... call continues on hold ...
Customer re-INVITE (SDP with a=sendrecv) → Telnyx  ← Taking off hold
OK ← Telnyx → Customer
```

Re-INVITEs modify sessions mid-call. On-hold uses `a=sendonly` (I send you media, don't send me any). Resume uses `a=sendrecv`. Failure here causes one-way audio after hold/resume.

## Real-World Troubleshooting Scenario

**Customer report:** "Calls to some numbers get 408 timeout, but other calls work."

**Investigation:**

1. Find a failed call trace in Graylog:
```
type:SIP AND status:408 AND destination:+15559876543
```

2. Examine the flow:
```
14:00:00.123 INVITE (customer) → sip-proxy (inbound)
14:00:00.456 INVITE (sip-proxy) → carrier (outbound)
14:01:00.789 408 Request Timeout (sip-proxy) ← carrier (outbound)
14:01:00.790 408 Request Timeout (customer) ← sip-proxy (inbound)
```

3. Check which carrier was used:
- The outbound INVITE shows the carrier IP/domain

4. Compare to a working call to a different number:
- Same time period, different carrier
- That call completed successfully

5. Root cause: Specific carrier (let's call them "Carrier-X") is not responding to INVITEs. The 408 is coming from our timeout, not from the carrier.

**Mitigation:** Route around Carrier-X via configuration change → calls succeed.
**Resolution:** Escalate to Carrier-X's operations team about their unresponsiveness.

**Time to root cause with trace:** 5 minutes
**Without trace:** Would have required VPN testing, guessing, hours of investigation

---

**Key Takeaways:**

1. SIP traces are logged to Graylog, indexed by Call-ID, phone number, timestamp, and connection
2. Traces show interleaved messages from customer leg (inbound) and carrier leg (outbound)
3. Systematic reading: Find error → Identify leg → Read triggering request → Check auth → Examine SDP
4. 401/403: Authentication/authorization issues; check for credential mismatch or authorization failure
5. 408: Timeout — usually means no response from far end (carrier down, network issue)
6. Codec mismatches in SDP cause 488 or no audio — trace the offer/answer exchange
7. Mid-call re-INVITE failures cause hold/resume problems — one-way audio after transfer/hold
